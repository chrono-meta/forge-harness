# convergence-loop — 범용 수렴 루프 게이트 강화

> 단 한 번의 통과 선언은 신뢰하기 어렵다. Fix가 새 FAIL을 노출하고, 1라운드가 놓친 것을 2라운드가 잡는다. convergence-loop은 어떤 게이트에든 "진짜 통과" 기준을 부여한다.

## 발굴 경위

두 곳에서 동일 패턴이 반복 등장하면서 추출됨 (2026-05-23):
- `mcp-spec-to-tc` Step 4.5 P3 게이트: 1라운드 ✅ 후 2~3라운드에서 추가 FAIL 발견 → 수렴 루프 3라운드 내재화
- `harvest-loop` Step H1~H5: 패턴 추출 → 공격 → 합성 → 반복 구조가 수렴까지 돌아야 함

같은 구조가 두 번 나오면 스킬이다.

## 적용 대상

| 적용 가능한 게이트 | 예시 |
|---|---|
| TC 품질 게이트 | mcp-spec-to-tc P3 9/9 + Wave-P3 |
| 스킬 진단 게이트 | harness-doctor + apex-review |
| 세션 수확 루프 | harvest-loop Step H1~H5 |
| 외부 자산 감사 | steel-quench Wave 1~N |
| 어떤 FAIL→FIX 반복 구조든 | 사용자 지정 게이트 |

## 파이프라인 구조

```
[입력] 게이트명 + 판정 기준 + 최대 라운드 N (기본 3)
    │
    ▼
라운드 1
    │  게이트 실행
    │  → 전항목 통과: ✅ 1라운드 통과 → 라운드 2 (검증)
    │  → FAIL 발생: FAIL 항목 명시 → FIX 실행 → 라운드 2
    │
    ▼
라운드 2
    │  동일 게이트 재실행 (FIX 반영 + 신규 FAIL 탐색)
    │  → 전항목 통과: ✅ 2라운드 통과 → 라운드 3 (최종 확인)
    │  → 신규 FAIL: 명시 → FIX → 라운드 3
    │
    ▼
라운드 3 (최종)
    │  → 전항목 통과: ✅ 진짜 통과 선언
    │  → FAIL 잔존: "구조적 재설계 필요" → 에스컬레이션
    │
    ▼ (N라운드 내 미수렴 시)
에스컬레이션
    → 원인 분류: 판정 기준 모호 / FIX 역량 한계 / 게이트 설계 결함
    → 권장 조치 출력
```

**핵심 원리**: Fix가 새 FAIL을 노출하는 것은 실패가 아니라 **게이트가 제대로 작동하는 신호**다. 라운드가 깊어질수록 발굴되는 FAIL은 더 근원적이다.

## 실행 지침

### 설정 (실행 전 확인)

```
게이트명:     [어떤 체크포인트인가]
판정 기준:    [통과 조건 명시 — "전항목 ✅" 또는 구체 기준]
최대 라운드:  [기본 3, 단순 게이트는 2, 복잡 게이트는 5 가능]
FIX 주체:    [자동 FIX 가능 / 사람 개입 필요 / 혼합]
에스컬레이션: [N라운드 미수렴 시 누구에게 / 어떻게]
```

### 라운드별 실행 규칙

**라운드 간 유지해야 할 것**:
- 판정 기준 불변 (라운드마다 기준 완화 금지)
- FAIL 항목 누적 추적 (이전 라운드 FAIL이 FIX됐는지 확인)
- 신규 FAIL 명시 (FIX 연쇄로 새로 드러난 것 구분)

**라운드 간 바뀌는 것**:
- FIX 반영된 대상 (수정된 TC·스킬·문서)
- FAIL 목록 (줄어야 정상, 늘어나도 수용 — 드러난 것)

### 수렴 판정

```
수렴 = 연속 2라운드에서 신규 FAIL 0건
진짜 통과 선언 조건:
  1. 전항목 통과 AND
  2. 최소 2라운드 실행됨 (1라운드 단독 통과는 "잠정 통과"로만 처리)
```

### 에스컬레이션 원인 분류

| 원인 | 신호 | 권장 조치 |
|---|---|---|
| 판정 기준 모호 | 같은 항목이 매 라운드 다르게 판정됨 | 판정 기준 재명문화 후 재시작 |
| FIX 역량 한계 | FAIL은 감지되나 FIX 방법을 모름 | 전문가 개입 또는 범위 축소 |
| 게이트 설계 결함 | N라운드 후에도 같은 FAIL 반복 | 게이트 자체 재설계 → meta-prompt-builder |

## 출력 형식

```
## convergence-loop 실행 결과

게이트: [게이트명]
최대 라운드: N | 실제 수렴 라운드: M

| 라운드 | 판정 | FAIL 항목 | 신규 FAIL | FIX 실행 |
|:---:|:---:|---|:---:|:---:|
| 1 | FAIL/PASS | [항목 목록] | — | Y/N |
| 2 | FAIL/PASS | [항목 목록] | [신규] | Y/N |
| 3 | PASS | 없음 | 없음 | — |

✅ 진짜 통과 (M라운드 수렴)
❌ N라운드 미수렴 → 에스컬레이션: [원인 분류]
```

## 연계 스킬

| 상황 | 연계 스킬 |
|---|---|
| TC P3 게이트에 적용 | `mcp-spec-to-tc` Step 4.5 (내장 구현체) |
| 스킬 진단 게이트에 적용 | `harness-doctor` → convergence-loop 래핑 |
| 담금질 Wave에 적용 | `steel-quench` Wave 3+ 수렴 기준 동일 원리 |
| 세션 수확 루프에 적용 | `harvest-loop` Step H1~H5 |
| 게이트 재설계 필요 시 | `meta-prompt-builder` |

## Done When

```
설정 완료 (게이트명·판정 기준·최대 라운드 확정)
+ 최소 2라운드 실행
+ 수렴(전항목 통과 2라운드 연속) 또는 에스컬레이션 선언
+ 라운드별 결과 표 출력
```
