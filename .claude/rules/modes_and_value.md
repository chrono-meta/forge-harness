# Harness Usage Modes + Differentiated Value

## Usage Modes вЂ” "Don't block those who come, don't block those who leave"

The forge-harness maintains **a reference asset identity** вЂ” user entry and exit are free. The AI guides and supports any mode without refusal.

### 3 Mode branches

| Mode | Essence | AI guidance and support obligation | Contribution path |
|---|---|---|---|
| **A вЂ” Canonical** | Harness cwd setup в†’ hand off to separate project cwd в†’ field agent works | Active onboarding 5-skill cascade + cwd handoff guidance | Direct (bidirectional sync circuit active) |
| **B вЂ” Resident** | Create a **separate project directory** in the harness install environment and work there | Required to guide "separate project directory + proper `.gitignore`" | Indirect (harness itself maintains reference asset identity) |
| **C вЂ” Plugin/skill only** | Install only plugin/skill without cloning harness | Guide user to accumulate history on their own project side. Explicitly state no automatic harness signals expected | Indirect + dependent on user active invocation |

### AI guidance principles

- If user explicitly states a mode, immediately guide that mode. Do not force canonical mode (Mode A).
- Mode B: **Mandatory check of separate project directory creation + `.gitignore` harness asset isolation**
- Mode C: Guide history accumulation mechanism on user's project side. Harness side actively absorbs via issue monitoring and PR audit cadence.
- In any mode: do not accumulate user personal work in the harness directory itself (protect reference asset identity)
